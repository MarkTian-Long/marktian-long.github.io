# 用 Claude Code 重设计个人品牌网站：从求职主页到 Editorial 风格思想站

> 写于 2026-04-12。这是一篇过程记录，不是教程，包含思路、决策、踩过的坑。想以后精简发出去。

---

## 背景与起点

我的个人网站最初是一个典型的 AI 求职主页——深色科技风，`#080c18` 背景配蓝紫渐变 Glow，满屏 badge 和 tag，视觉上像一个 AI SaaS 落地页，而不是"一个有立场的人写的东西"。

问题不在于丑，而在于**定位失准**：目标受众（AI 领域的技术负责人、CTO、同行 AI PM）在看这类页面时，真正关注的是"这个人思考的密度和独立性"，而不是视觉炫技。深色科技风反而发出了相反的信号——这是一个想装酷的人，而不是一个值得认真读的人。

重设计的核心目标：**让这个站看起来像一本你会认真读的独立杂志，而不是一份精心排版的简历**。

---

## 风格选择：为什么是 Claude Editorial

我用 Claude Code 从 62 个候选设计系统里筛选（通过 `brand-design-md` skill 生成的对比分析），最终的决策路径：

- 排除深色系（Spotify / Tesla / Vercel）→ 目标是浅色、可长期阅读的
- 排除消费品/金融感（Nike / Stripe）→ 定位不符
- 最终候选：Claude、Notion、Apple、Linear

**Claude 胜出的理由有两层：**

一是颜色系统本身。羊皮纸暖白 `#f5f4ed` + 陶土色 `#c96442`，形成"温暖知性的思想输出者"的感觉，不是冷峻的工具平台，也不是消费品的暖心。

二是信号价值。对于 AI 领域的目标受众，"这个站用的是 Claude 的美学语言"本身就是一个品味信号——不是刻意为之，而是用了之后自然带来的。

## 设计约束系统：impeccable

这次重设计有一个关键工具：[impeccable](https://github.com/pbakaus/impeccable)，一个为 LLM 辅助设计设定硬性约束的 skill 系统。17 个 skill，7 个参考领域，覆盖字体、颜色、空间、动效、交互、响应式、UX 写作。

**安装方式：**
```bash
# 项目级（优先）
cp -r impeccable/ .agents/skills/

# 用户级（跨项目复用）
cp -r impeccable/ ~/.claude/skills/
```

最关键的**绝对禁令（absolute bans）**：

| 禁止 | 原因 |
|------|------|
| `border-left/right > 1px` 彩色边框 | 产生"卡片化"的后台管理感，破坏 editorial 气质 |
| gradient text（`background-clip: text` + 渐变） | AI slop 的典型特征，立刻失去可信度 |
| bounce / elastic 动效 | 消费品感，与知性定位冲突 |
| 纯黑 `#000` / 纯白 `#fff` | 对比太强，失去温度 |

这些约束在实际执行中多次救了我。比如原计划在 Writing 区列表用 `border-left: 2px solid var(--accent-clay)` 做 hover 效果，被 impeccable 的 ban 1 拦住，改为 hover 时背景色变为 `--bg-secondary`，反而更克制更对。

## 字体系统升级

原来用 Georgia + Inter。Georgia 过于通用（全网泛滥），Inter 被 impeccable 列为需要审慎使用的字体。

**新方案：Libre Baskerville + Source Sans 3**

- **Libre Baskerville**（衬线 display）：字重对比强，书本质感，italic 字形有个性。用于大标题、序号、日期、宣言句。
- **Source Sans 3**（人文主义无衬线）：Adobe 出品，比 Inter 更有温度，字母形态更有设计感。用于正文和 UI 文字。

**字体的使用逻辑：**
- 衬线字体 = 有立场的内容（标题、日期、斜体宣言）
- 无衬线字体 = 功能性信息（导航、描述、标签）

这个分工让页面有了明确的信息层次，不是靠颜色区分，而是靠字体"性格"区分。

## Hero 区：从求职自我介绍到有冲击力的首屏

**改动前的问题：**
- 标题居中，"你好，我是 Mark，AI 产品经理"，一眼看出是简历站
- 副标题三行，每行 `<br>` 分隔，读起来像 PPT bullet point
- 没有任何"这个人有立场"的信号

**改动后的思路：**

1. **左对齐**：editorial 不居中。把所有内容左对齐，视觉重心从"我要展示自己"变成"我在叙述"。

2. **大标题单行**：`你好，我是 Leo`，斜体陶土色 `Leo`。Libre Baskerville italic 的字形在这个字号下有很强的个性，陶土色既是品牌色又是"这是重点"的信号。

3. **副标题缩为一句**：原来三行列举变为一句完整表述。三行的问题是在罗列而不是在表达，读者扫一眼感受不到判断力。

4. **宣言句**：`不写最优解，只写我真实认为的。` 用 Libre Baskerville italic，陶土色，低 opacity。这句话是整个站的立场宣言，放在副标题之后、按钮之前，读者会在行动前先看到它。

5. **装饰字母 L**：在 hero 背景右侧放了一个极低 opacity（0.045）的 Libre Baskerville italic 大写 L，字号 `clamp(20rem, 38vw, 32rem)`。这是 editorial 风格的经典手法——隐约的纸质质感，不是功能元素，是氛围。

## Judgment 区：从折叠卡片到 Editorial 观点列表

这是改动最大、迭代最多的区块。

**原版问题：** 一个大型折叠卡片，里面嵌套观点列表 + 行业网格 + 能力短板，视觉上像企业后台的"模块详情展开"，而不是一个人的判断表达。

**关键设计决策过程：**

### 决策 1：去掉外层折叠卡片

原来整个判断区是一个可折叠的 `.ai-judgment` 卡片，有橙色顶部边框（违反 impeccable ban 1 的边境边框感）。

改为直接展示，5条观点以大序号列表呈现，没有外层容器。理由：折叠本身是在藏内容，但这个区块的内容是整个站的核心价值——为什么要藏？

### 决策 2：序号大于内容

观点列表用 Libre Baskerville 大号序号（01-05），默认 opacity 0.25，hover 或展开时跳到 0.8。这个设计的逻辑：

- 平时克制（0.25），不抢夺正文注意力
- 交互时响应（0.8），给用户明确的"你点到了这里"的信号
- 衬线大号数字本身就是 editorial 风格的经典排版手法

### 决策 3：行业落地区块的逻辑重构

原版用网格展示 7 个行业，但发现问题：客服/售前、软件开发辅助、内容创作是**功能场景**，金融/教育/医疗/法律是**行业**——同一个网格里混了两个维度，类比逻辑失效。

最终改为两级结构：
- **第一层**：通用场景横排标签（凡标准化程度高、错误代价低，AI 都跑通了）
- **第二层**：行业进度列表（各行业由自身壁垒决定，而非技术成熟度）

这个结构的好处不只是视觉，更是**论证逻辑更清晰**——读者看完能理解"为什么客服 AI 已规模化但医疗还早期"。

### 决策 4：展开链接的可见性

展开后的"思考碎片"和"工具"链接，最初是纯文字陶土色，读者不容易感知可以点击。

改为参考 `index.html` 里已有的 `expand-btn-link` 样式：小按钮轮廓，加类型前缀（`📝 思考碎片` / `🛠 工具`）。按钮轮廓 + 类型标注 = 读者一眼知道"这是个可以点的东西，点了会去哪里"。

## 颜色策略：双轨

这次重设计确立了明确的**双轨颜色策略**：

| 颜色 | 用途 | 含义 |
|------|------|------|
| 陶土色 `#c96442` | section-label、观点序号、日期、标签、展开链接 | 思想性元素 |
| 蓝色 `#2563eb` | 主要 CTA 按钮、外链 | 行动性元素 |

这个区分的本质是：**想让读者思考的地方用陶土色，想让读者点击的地方用蓝色**。不混用，不随意。

## 个人定位的表述迭代

从 `AI 产品经理` 到 `AI · Product · Builder`，经过了几轮讨论：

**问题**：`AI 产品经理` 这个标签准确但单薄——实际上我也持续在写代码、构建工具，这部分信息被完全遮蔽了。

**迭代过程：**
1. `AI Product Manager` → 过于职能化
2. `AI & Product` → 太模糊
3. `AI · Product · Builder` → 三词各有指向：AI（研究跟踪）、Product（产品决策）、Builder（动手构建）

**为什么选英文而不是中文"AI 产品与工程实践"：**
目标受众是技术圈，`Builder` 这个词在这个圈子里有明确的身份共鸣——它意味着"不是只动嘴的，是真的在做的"。中文版本失去了这个信号密度。

三词用 `·` 分隔而不是空格，是因为 `·` 在 editorial 排版中是节奏符，比空格更有视觉停顿感。

## 联系方式的设计细节

**Phone / WeChat 合并为一个条目**

微信号和手机号相同，本来是两个独立条目，但读者会疑惑"为什么要显示两次同一个号码"。

合并后 label 写 `Phone / WeChat`，一个条目承载双重信息，点击复制号码。更简洁，没有歧义。

## 工具使用经验

### Claude Code 在 UI 改动中的边界

**最有效的场景：**
- 精确的 CSS 变量替换和新样式编写
- HTML 结构重构（给出明确的新结构描述）
- JS 交互逻辑（折叠/展开、复制等简单功能）

**需要人工把关的场景：**
- 分类逻辑是否正确（行业网格的维度混用就是 AI 没有主动发现的）
- 视觉节奏和阅读体验（截图反馈比文字描述有效 10 倍）
- 定位文字的表达（"Builder" vs "工程实践"这类判断 AI 会给建议但你需要拍板）

### 截图反馈比文字描述有效

整个过程中，提供截图 + 红框标注的方式比文字描述问题快很多。AI 能从截图里直接识别"那个蓝色方框链接"是 focus outline 还是 CSS 样式，而文字描述"链接有点奇怪"无法给 AI 足够的信息。

### impeccable 的价值

不是在改完之后 review，而是**在改之前就知道什么不能做**。这个思维转变很重要——与其让 AI 自由发挥然后一起改，不如先建立约束边界再在边界内工作。

## 教训和反思

### 1. Preview 文件的管理

整个重设计用 `index-preview.html` + `style-light.css` 作为沙盒，正式文件保持不动，最后确认后再替换。这个做法是对的，但中间版本积累过多，有几次差点搞混"当前改的是哪个版本"。

**改进建议**：下次用 git branch 做隔离，不要靠文件名区分版本。

### 2. CSS 清理的惰性

改动过程中有大量旧 CSS 类（`.ai-judgment`、`.prediction-*`、`.industry-card` 等）变成孤儿——HTML 里不再使用，但 CSS 里还留着。这些死代码会让后续维护者（包括未来的自己）困惑。

**改进建议**：每完成一个大改动，立刻搜索并删除孤儿 CSS，不要攒到最后。

### 3. 分类逻辑要在改动前确认

行业落地网格的维度混用问题（功能场景 vs 行业），是在做完之后看截图才发现的。如果在写 HTML 之前先把内容列出来确认逻辑，可以省掉一轮返工。

**改进建议**：内容性区块（不是纯样式）改动前，先把内容结构用纯文字写出来确认，再写 HTML。

### 4. 移动端同步验证

大部分改动只在桌面端验证，移动端适配是事后补的。有几处（如 `view-head` 的缩进计算）在移动端需要额外调整。

**改进建议**：关键改动后立刻切 DevTools 移动端视图确认，不要攒到最后。
