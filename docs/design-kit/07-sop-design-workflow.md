# 设计 → 实现 SOP 流程

## 一、设计前：确定上下文

在写任何代码前，必须确认：

| 问题 | 说明 |
|------|------|
| 目标用户是谁？ | 角色、使用场景、需要完成什么任务 |
| 品牌性格是什么？ | 用 3 个具体词形容（不要"现代/优雅"这种死类别） |
| 界面应该给人什么感觉？ | 信任/轻松/专注/紧迫/愉悦？ |
| 亮色还是暗色主题？ | 根据场景推导，不要凭习惯 |
| 有没有参考/反参考？ | 参考什么的哪个具体细节，明确说出来 |

这些不能从代码里推导，只能问人/从需求文档里找。

## 二、设计决策顺序

```
1. 色彩方向（主题 + 调色板结构）
      ↓
2. 字体选型（显示字体 + 正文字体）
      ↓
3. 布局结构（网格、容器、间距节奏）
      ↓
4. 组件设计（按钮、卡片、导航等）
      ↓
5. 交互状态（hover/focus/active/disabled）
      ↓
6. 边界状态（empty/loading/error/overflow）
      ↓
7. 动效（入场/过渡/反馈）
      ↓
8. 响应式适配（不只是缩小，要重新设计）
```

## 三、实现时的检查习惯

### 每个组件完成时检查：
- [ ] 所有颜色都用了 CSS 变量，无硬编码
- [ ] hover/focus/active 三个状态都有视觉反馈
- [ ] 空状态不只是"没内容"，要有引导信息
- [ ] 文字行宽不超过 65-75ch
- [ ] 字号层级清晰，同级字号不要差 1-2px

### 布局完成时检查（眯眼测试）：
- 眯起眼睛模糊看页面，最重要的元素是否仍然最显眼？
- 每个区块的重要性是否通过大小/间距/颜色同时体现？
- 有没有"全部都一样大"的问题？

### 完成后用此问题自测：
> 如果给别人看这个界面说"这是 AI 做的"，他们会立刻相信吗？  
> 如果会 → 设计还不够有意图，继续迭代。

## 四、常见 AI 审美陷阱（务必避免）

### 立即识别并重写的 CSS 模式

**1. 侧条纹卡片**
```css
/* 禁止 — 最常见 AI 设计痕迹 */
border-left: 3px solid var(--color-warning);
border-left: 4px solid oklch(...);
```
替代方案：全边框 / 背景色块 / 序号/图标前缀 / 什么都不加

**2. 渐变文字**
```css
/* 禁止 */
background: linear-gradient(...);
-webkit-background-clip: text;
-webkit-text-fill-color: transparent;
```
替代：单一纯色文字。强调改用字重/字号/颜色

**3. 毛玻璃到处用**  
毛玻璃（backdrop-filter: blur + 透明背景）只在**有实际意义**时使用，不是装饰手段

**4. 每个标题前都放大圆角图标**  
图标 + 圆角矩形容器如果不增加信息密度，就是模板化痕迹

**5. 英雄指标布局**  
`大数字 + 小标签 + 辅助统计 + 渐变强调`——已经是审美公共品

### 同样要避免的布局模式
- 所有卡片等宽等高同结构（自动格栅）
- 所有间距相同（没有节奏）
- 全部居中对齐（左对齐更有设计感）
- 卡片嵌套卡片

## 五、响应式适配原则

不是把桌面版"缩小"，而是为移动端**重新设计**：

| 桌面 | 移动 |
|------|------|
| 双栏布局 | 单栏堆叠 |
| 宽导航栏 | 汉堡菜单 |
| 悬浮 hover 交互 | 改为 active/tap 反馈 |
| 大量信息并排 | 折叠/手风琴 |
| 大字号标题 | 等比缩减（`clamp()` 处理） |

**移动端必须可用**，不能只靠 `display: none` 隐藏功能。

## 六、动效使用规则

只做**两件事**的动画：**transform** 和 **opacity**。其他属性会触发 layout/paint，导致卡顿。

```css
/* 正确 */
transform: translateY(-4px);
opacity: 0.8;

/* 错误 */
height: 0 → auto;   /* 改用 grid-template-rows 动画 */
width: 200px;
padding: 0 → 16px;
```

**easing 选择**：
- 元素进入 → ease-out（先快后慢，模拟物理降落）
- 元素离开 → ease-in（先慢后快，轻轻离去）
- 状态切换 → ease-in-out
- 禁止 bounce / elastic（2015 年就过时了）

**时长**：UI 反馈 ≤ 300ms，布局变化 300-500ms，入场动画 500-800ms。

## 七、代码实现顺序

1. **HTML 语义结构**（先不写任何 CSS）
2. **布局和间距**（建立视觉节奏）
3. **字体和颜色**（应用 token 系统）
4. **交互状态**（hover/focus/active）
5. **边界状态**（empty/error/loading）
6. **动效**（最后加，不是一开始就做）
7. **响应式**（验证每个断点）

## 八、设计意图记录

每个非显而易见的设计决定都要在 PR 描述里说明原因，方便后续维护：

- 为什么用陶土色做标签而不是蓝色？（区分行动性/思想性语义）
- 为什么用衬线字体做大标题？（对比正文无衬线，增加文化厚度）
- 为什么英雄区用背景大字而不是图片？（避免维护图片资源，用字体即可）
