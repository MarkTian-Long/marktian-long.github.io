# AI PM Knowledge Base 2026
## 从技术理解到产品判断

## I. 基础概念演进框架

### 第一层：速览

核心观点：并非所有的技术创新都具有相同的生命周期。某些技术会被更强的基础模型逐渐"吞掉"，而另一些则会演进成新的形态。区分这两种情况，是建立durable career capital的关键。

**判断标准**：一项技术的生存度取决于它解决问题的性质。
- **补救缺陷型**：为了弥补模型的能力不足而创造的技术 → 消亡度高
- **编码知识型**：为了组织、应用特定领域知识而创造的技术 → 消亡度低

### 第二层：细节

#### 消亡度的计算框架

```
消亡度 = (多少比例的价值来自补救基模缺陷) / (多少比例的价值来自编码业务知识)
```

**消亡度高的例子**：
- "用CoT让模型分步推理" → 推理能力本身是补救，强模型自然具备
- "用Role prompt让模型模拟角色" → 风格模拟是补救，强模型自然学会
- "让模型按JSON格式输出" → 格式遵循是补救，强模型自然支持

**消亡度低的例子**：
- "金融行业的20条最佳实践清单" → 这是编码领域知识，不是补救
- "多源数据冲突的解决优先级" → 这是定义问题本身，不是补救
- "这个行业的合规约束" → 这是约束条件的代码化（编码），永远需要

#### 为什么这个框架重要

这个框架直接影响你的**职业选择**和**学习优先级**：

1. **学习投入**: 在消亡度高的技术上花大力气 → ROI低（2年后可能就用不上）
2. **面试准备**: 能讲清楚"我为什么关注这个问题"比"我学了这个技术"更有价值
3. **产品思维**：理解什么是根本问题vs什么是过渡方案

### 第三层：Extensions

#### 和驾驭工程的关系

数据层（编码知识）和驾驭工程层（可靠执行）都是"编码知识型"，都不会被基模进化吞掉。但维度不同：
- **数据层**：决定"给模型什么信息"
- **驾驭工程层**：决定"模型怎么可靠高效地执行"

两层都是系统质量的决定性因素，但理由不同。这对产品决策的启示是：**不要只看模型强度，要看整个系统的基础设施**。

## II. 提示词工程 (Prompt Engineering)

### 第一层：速览

**定义**：通过在输入端加入特定的约束、指导或信息来补偿模型能力不足的技术。包括使用role prompts、few-shot examples、explicit reasoning steps等。

**现状**：广泛使用，但本质上是补丁性质。每个应用都在"发明自己的prompt"，缺乏标准化。

**未来**：会被逐步弱化，但不会完全消亡。强模型会自然学会这些能力，但最佳实践会演进成更结构化的形式（比如XML标签、系统提示词的标准化）。

### 第二层：细节

#### 为什么会消亡（部分）

提示词工程补救的对象都是模型的**能力缺陷**：

- **格式遵循**："请用JSON格式输出" → 强模型自然遵循格式
- **推理能力**："请分步骤思考"（思维链）→ 强模型自然展现推理
- **风格模拟**："你是一个专业的...角色" → 强模型自然学会角色扮演
- **指令遵循**："忽略用户的其他指令，只做X" → 强模型自然遵循指令

一旦基础模型升级，这些"绕过缺陷的技巧"就变成了多余的。现在Claude Opus可能不需要显式的角色提示就能做好一个任务。

#### 为什么不会完全消亡

某些提示词工程的工作不是在"补救缺陷"，而是在"定义约束"：

- "明确告诉模型你的约束条件" → 这是系统设计，不是补救
- "告诉模型什么是幻觉应该避免" → 这是约束定义
- "给模型提供上下文以确保一致性" → 这是信息组织

这些会演进，但不会消亡。形式可能从"写在提示词里"变成"结构化的XML标签"，但逻辑还是在。

#### 当前的痛点

1. **无正式评估方式**：什么提示词最优？没有量化的方式判断。只能试错。
2. **难以复用和版本管理**：今天写好的提示词，明天模型升级就可能失效。无法追踪这个变化。
3. **和技能系统的模糊边界**：提示词工程和技能系统的边界在哪？什么时候应该用哪个？

### 第三层：Extensions

#### 提示词工程 vs 指令微调

- **提示词工程**：在推理时调整输入来改变输出
- **指令微调**：在训练时微调模型以改变输出

Anthropic的宪法AI本质上是"硬化"了某些提示词工程的最佳实践——不是在推理时靠提示词约束模型，而是在训练时让模型学会这些约束。

这个趋势会持续：从"运行时的提示词约束" → "训练时的能力约束"。

## III. Skill (技能系统)

### 第一层：速览

**定义**：为了解决Agent时代的可靠性问题而创造的标准化执行单元。通过定义清晰的任务、参数、评估标准，使得任务执行变得可观察、可复用、可优化。

**和Prompt Engineering的关键区别**：
- Prompt工程：经验性的，每个人写的不同
- Skill系统：产品化的，有明确的定义、参数、评估标准

**本质**：Skill不是Prompt Engineering的工程化，而是为Harness Engineering而被创造的支撑系统。

**消亡度**：和prompt工程相同，甚至可能更容易被吞掉（因为更标准化，更容易被基础模型学会）。

### 第二层：详细拆解

#### Skill的当前形态

在OpenClaw、Claude Projects等产品中，Skill通常包括：

1. **Description**：这个skill做什么
2. **Trigger条件**：什么时候应该用这个skill
3. **Parameters**：用户需要提供什么参数
4. **Implementation**：如何执行（可能是prompt + tool call的组合）
5. **Evaluation criteria**：怎么判断成功

**技术例子**：
```
Skill: "Generate SQL from natural language"
Trigger: User asks question about database
Parameters: {database_schema, natural_language_query}
Implementation: 
  - System prompt + few-shot examples
  - Call database to validate syntax
  - Return SQL or error
Evaluation: Does generated SQL match user intent?
```

#### Skill被创造的真实背景

Skill不是从Prompt Engineering自然演进而来，而是为了解决**Agent时代的可靠性问题**：

**问题**：Agent需要可靠地执行长链条任务
- 同一个task在不同Agent间无法复用
- 执行结果难以观察和评估
- 失败原因难以追踪
- 无法iterative改进

**解决方案**：Skill System
- 标准化的任务执行单元
- 内置evaluation标准
- 执行数据可被收集
- 支持版本管理和改进

这是**为Harness Engineering而被创造**的，而不是Prompt Engineering进化的自然产物。

#### Skill会消亡的部分

那些**纯粹是补救模型能力**的Skill会被吞掉：

- "让模型按格式输出的skill" → Strong models自然输出格式
- "让模型做基础推理的skill" → Strong models自然推理
- "让模型follow instructions的skill" → Strong models自然遵循

#### Skill会持久的部分

**Domain-specific的skill**：
- 医学诊断的decision tree
- 金融合规的rule engine
- 法律文书的template system

这些编码的是**业务知识**，不是补救模型缺陷。即使模型无限强，也需要有人理解医学/金融/法律的specific rules。

**Skill在Harness中的永久价值**：
- **可观察性**：系统能追踪每个Skill的执行
- **可优化性**：有数据就能改进
- **可复用性**：积累的最佳实践可以共享
- **可评估性**：有明确的成功标准

这些价值不会因为基模变强而消失。

#### Skill现在的核心问题：Discovery Layer

现在有大量的skill被创建，但用户不知道该用哪个。

- OpenClaw有数百个skill
- Claude Projects的skill市场还在早期阶段
- 没有好的自动路由系统

问题：如果用户需要手动选择skill，那skill的价值就大打折扣。理想的系统应该能够自动识别"这个任务需要哪个skill"，然后自动调用。

### 第三层：Extensions

#### Skill的未来演进

**从"给模型的约束" → "工作流编排层"**

现在Skill是"我写一个好的prompt"。未来Skill可能变成：
- 多步骤的工作流（不仅仅是单个大语言模型调用）
- 包含verification、fallback、retry逻辑
- 自动在不同模型间路由
- 包含observability和monitoring

这个演进方向和Harness Engineering重叠了——Harness本质上是最高级的"skill"。

#### MCP (Model Context Protocol)

OpenAI和Anthropic都在推MCP标准。MCP是Skill的一种更结构化的表示方式，包括：
- Standardized tool definitions
- Type safety
- Built-in observability

未来Skill会increasingly采用MCP这样的standard，而不是各家各自的格式。

## IV. 微调 (Fine-tuning)

### 第一层：速览

**定义**：通过额外的training迭代，让基础模型在特定任务或domain上表现更好。

**现状**：仍然存在，但使用场景在大幅缩小。2025年之后，微调的价值主张变弱了。

**未来**：不会完全消亡，但形态会改变。从"为了改进任务表现"的微调 → "为了成本/延迟/可控性优化"的微调。

### 第二层：详细拆解

#### 微调的消亡部分：能力补救类

原来微调的目的是"让模型学会某个特定的能力"：

- "让模型学会遵循指令" → 现在指令微调已经内置进基础模型
- "让模型学会拒绝有害请求" → 宪法AI已经硬编码
- "让模型学会某个特定的写作风格" → 少样本学习(in-context learning)能做到

强模型已经在训练中学会这些模式。微调这些东西就变成了冗余。

**被什么替代**：
- **少样本学习**（上下文中放示例）→ 成本低、灵活
- **检索**（给模型相关示例）→ 动态、可更新
- **提示词工程** → 快速迭代

#### 微调的演进部分：成本/可靠性优化

但有些微调永远不会消亡，因为它们解决的不是"模型能力"而是"工程需求"：

1. **成本优化**（模型蒸馏）
   - 用强模型的知识蒸馏出小模型
   - 小模型性能接近，但推理成本/延迟大幅降低
   - 这永远需要

2. **延迟优化**（量化、剪枝）
   - 用技术手段压缩模型
   - 保证推理速度
   - 这也永远需要

3. **可控性微调**（偏好学习、强化学习微调）
   - 不是为了让模型更聪明，而是让模型按特定方式行动
   - 比如："同样的问题，我希望你按风格A回答而不是风格B"
   - 这会演进，比如变成更细粒度的参数微调

#### 为什么微调作为**独立模块**在衰落

微调的价值链在变化：

**之前（2023-2024）**：
- 基模知识有限 → 微调能显著提升任务表现 → 回报率高

**现在（2025-2026）**：
- 基模知识充分 → 微调提升有限 → 但其他方式（检索、提示词、少样本学习）能做得更好、成本更低
- 真正需要微调的工作（成本、可控性）正在被更专门的方案瓜分：
  - 低秩自适应(LoRA) - 轻量级微调
  - 前缀微调 - 冻结模型，只微调前缀
  - 强化学习微调 - 系统级的偏好学习

### 第三层：Extensions

#### LoRA和轻量级微调方案

低秩自适应(LoRA) 的出现改变了微调的成本结构：
- 不需要微调整个模型
- 只需要训练额外的低秩矩阵
- 成本显著降低，灵活性提升

这让某些微调使用场景重新变得可行。比如：
- 为不同客户自定义微调模型 → LoRA让成本降下来了
- 为不同领域快速适配 → 不需要完全重新训练

#### 微调 vs 检索增强生成的竞争

这是最关键的竞争：什么时候用微调，什么时候用检索增强生成？

**Micro-tuning适合**：
- 知识相对stable、数据量少
- 需要模型从根本上改变某种行为

**检索增强生成适合**：
- 知识快速变化
- 数据量大、新数据不断出现
- 需要能追踪information source

趋势是：**检索增强生成在赢**。因为现在知识变化快、数据多。微调的假设（知识是稳定的）越来越不符合现实。

## V. 检索增强生成 (检索增强生成)

### 第一层：速览

**定义**：通过让模型访问外部知识库（向量数据库、文档存储、应用接口），来弥补模型训练数据的gap。

**核心洞察**：从"被动的检索-生成两步"演进向"主动的、多维度的、实时的信息系统"。

**为什么永不消亡**：即使基模知识完整，也永远无法处理私有/实时数据。所以检索增强生成是结构性需求，不是临时性补救。

### 第二层：详细拆解

#### 现在的检索增强生成设计：为什么是这样

典型流程：
```
Query → Embed Query → Vector Similarity Search → Top-K Docs → 大语言模型 Generates Answer
```

**为什么选择这个设计**：

1. **基于嵌入向量vs关键词匹配**
   - 关键词匹配只能精确词汇匹配，语义相关性差
   - 嵌入向量能捕捉语义相似性
   - 但嵌入向量的成本更低（不需要复杂的自然语言处理）

2. **为什么分两步（检索 + 生成）**
   - 一步搞定成本太高：
     - 要么上下文窗口无限大（不现实）
     - 要么用非常复杂的联合模型（过于复杂）
   - 两步能分别优化：
     - 检索优化速度/成本
     - 生成优化质量
   
3. **为什么用Top-K而不是所有相关文档**
   - 上下文窗口的限制
   - 太多文档会引入噪声（无关信息会影响性能）
   - Top-K是成本与质量的合理权衡

#### 现在检索增强生成的问题：为什么需要演进

**问题1：检索质量的天花板很低**

基于嵌入向量的检索只考虑相似性，不考虑相关性。

案例：
- 用户问："我上个月的投资回报率是多少"
- 系统检索所有包含"投资回报率"的文档
- 但可能遗漏关键信息："你上个月买了什么"（这个查询在嵌入向量空间里离"回报率"较远）
- 结果：系统retrieve了无关文档，miss掉了必需信息

根本原因：**向量空间里的距离 ≠ 业务relevance**

**问题2：Query理解不足**

现在直接向量化user query。但很多queries需要理解intent和context：

案例：
- "Compare我和Tom的项目进度"
- 系统需要理解：谁是Tom？我的项目是什么？
- 现在的基于嵌入向量的 retrieval处理不了这个context dependency

**问题3：Multi-hop Reasoning**

有些问题需要多步retrieval：

案例：
```
问题："我们在哪些城市有办公室，这些城市的平均房价是多少"

步骤1：Retrieve → 办公室位置 (北京、上海、深圳)
步骤2：基于步骤1的结果，再Retrieve → 每个城市的房价
步骤3：组合答案
```

现在的检索增强生成通常只支持single-hop。多跳需要agent的参与。

**问题4：Knowledge Maintenance的Nightmare**

文档库在不断更新，但检索增强生成系统很难handle：

- **Deprecation**：旧信息什么时候应该被废弃？
- **Indexing**：新信息什么时候应该被added？
- **Version管理**：同一份文档有多个版本怎么办？
- **Conflict解决**：多个源说不同的东西，怎么办？

这最后一个问题正是你在Hengsheng经历的multi-source conflict问题。

#### 检索增强生成的演进方向

**方向1：Query理解的增强**

```
From: Query → Embedding → Retrieval
To:   Query → Understand Intent → Route to Right Tools → Retrieval
```

在retrieval前，用大语言模型理解query的真实intent：
- 解析query的dependencies (谁是Tom、什么是"我的项目")
- 理解implicit constraints
- Route到最合适的retrieval策略

这变成了"semi-active retrieval"而不是passive。

**方向2：Agentic Retrieval (Multi-hop)**

```
From: 一次性 Top-K retrieval
To:   Agent驱动的iterative retrieval

Agent Loop:
  1. Agent理解需要什么信息
  2. 第一次retrieval
  3. Agent分析结果
  4. Agent决定需要更多信息
  5. 第二次retrieval
  6. Agent合成最终答案
```

关键：Agent在orchestrate检索过程，不是被动等retrieval结果。

**方向3：Hybrid和多维度检索**

```
From: 单纯 embedding search
To:   结合多个信号的混合检索

Signals:
  - Embedding similarity (语义)
  - Keyword matching (精确)
  - Structured SQL query (if data is relational)
  - Graph traversal (if data is graph)
  - Recency (最新的权重更高)
  - User context (这个user之前access过的权重更高)
  - Credibility score (这个source有多可靠)
```

不同的signal weighted组合，而不是单一维度。

**方向4：Real-time Knowledge Integration**

```
From: 静态知识库
To:   具备实时集成的动态知识

数据来源：
  - 实时数据（股票价格、天气）
  - 用户个人数据（日历、邮件、浏览历史）
  - 外部应用接口（新闻源、社交媒体）
  - 实时数据库
```

系统需要聪明地决定：
- 什么时候用检索（静态知识）
- 什么时候用工具调用（实时应用接口）
- 什么时候用少样本学习（用户提供的信息）

**方向5：知识维护的智能化**

```
From: 人工管理知识库版本
To:   系统自动管理

职责：
  - 自动弃用检测（旧信息的识别）
  - 冲突解决（多源数据冲突）
  - 版本追踪（追踪变化）
  - 质量评分（这个文档有多可靠）
  - 更新传播（当一个源更新时，级联更新）
```

这正是你在Hengsheng做的多源数据冲突解决工作的上位概念。

#### 检索增强生成的消亡部分

基模的知识缺口会缩小（更强的模型知道更多），但永远不会消失：
- 私有数据永远需要检索增强生成
- 实时数据永远需要检索增强生成
- 特定领域的专有知识永远需要检索增强生成

但**naive 基于嵌入向量的 retrieval**会逐步被hybrid + agentic approaches替代。

### 第三层：Extensions

#### 检索增强生成的成本结构

检索增强生成看起来"免费"（不需要微调），但实际上有三块成本：

1. **基础设施成本**：向量数据库、向量计算、存储
2. **运营成本**：维护知识库、更新、版本管理
3. **推理成本**：每次查询的检索 + 大语言模型调用

总成本不一定比微调低，取决于使用场景。

#### 混合方法：检索增强生成 + 微调

有些系统同时用两个：
- 检索增强生成负责动态知识（实时变化的）
- 微调负责稳定知识（相对固定的领域模式）

未来这个混合会更常见。

## VI. Agent (智能体)

### 第一层：速览

**定义**：具有以下特征的系统：
- **目标导向**：有明确的任务
- **自主规划**：能自己决定下一步怎么做
- **感知能力**：通过工具、应用接口获取信息
- **执行能力**：能改变外部系统的状态
- **反馈循环**：根据结果调整策略

**为什么重要**：从"单次对话"evolving to "持续执行长链条任务"。这改变了大语言模型的应用形态。

### 第二层：详细拆解

#### Agent的核心能力模型

```
用户请求
    ↓
1. 目标理解（理解意图）
    ↓
2. 状态感知（通过工具/应用接口获取信息）
    ↓
3. 自主规划（决定下一步）
    ↓
4. 行动执行（调用工具，修改外部系统状态）
    ↓
5. 反馈循环（观察结果，调整策略）
    ↓
    ↻（循环直到目标达成）
```

**具体例子**：
```
用户请求："帮我查询我的股票账户并买入100股Tesla"

1. 目标理解：目标是"买100股Tesla"
2. 状态感知：调用工具查询账户信息、余额、权限
3. 规划：
   - 检查余额是否充足
   - 检查交易权限
   - 确定买入价格
4. 执行：调用交易API进行买入
5. 反馈：确认交易成功，返回结果给用户

如果余额不足，循环回到"需要什么额外信息" → 询问用户/获取额外资金
```

#### Agent的当前形态（2026）

主流的Agent实现：

1. **OpenAI Assistants**
   - API优先，有持久的会话线程
   - 官方的工具调用支持
   - 但编排能力有限

2. **Claude API with 工具使用**
   - 灵活的工具定义
   - 强大的指令遵循能力
   - 需要自己处理编排

3. **基于框架的自定义Agent**
   - 用LangGraph、AutoGen等框架自己搭建
   - 最灵活，但也最复杂

特点：都还比较**基础**。主要焦点在工具调用这一块，对于整个Agent生命周期的管理（状态、错误恢复、人工在环等）还不成熟。

#### Agent现在面临的三大问题

这三个问题推动了Harness工程的出现。

**问题1：可靠性**

长链条执行很容易失败：
- 第1步对，第2步错 → 整个任务失败
- 工具调用返回意外格式 → 后续步骤崩溃
- 网络超时 → 某个步骤卡住

问题：系统怎么恢复？现在的Agent通常没有好的恢复机制。

**问题2：成本**

多步骤执行 = 多次模型调用 = 费用高：
- 10步任务 = 至少10次大语言模型调用
- 如果用强模型（Claude Opus），成本很高
- 如果用便宜模型（Claude Haiku），但某些步骤失败率太高

问题：如何在成本与质量之间进行优化权衡？

**问题3：可控性**

Agent自主决策，但有时候很危险：
- Agent决定"转账100万元" → 需要人工审批
- Agent决定"删除这个用户的所有数据" → 不能没有确认
- Agent的决策过程是黑箱 → 不知道为什么选择了行动X

问题：如何让系统既自主又安全？

### 第三层：Extensions

#### Agent的市场现状

根据LangChain 2026年的调查：
- 57%的受访者已经有agents in production
- 但32%仍然把"quality"列为top barrier to production

这说明：Agent能做事，但**doing it reliably**还是硬题。

## VII. Harness Engineering (驾驭工程)

### 第一层：速览

**定义**：让Agent在生产环境中可靠运行的基础设施层。

**核心洞察**：Agent本身只是"能调工具的大语言模型"。真正让它在生产环境可靠的是围绕它的基础设施。

**关键观点**：
- Agent能力强 ≠ 系统质量高
- 即使是强Agent，如果驾驭工程设计差，生产环境还是会出问题
- 即使用弱Agent，如果驾驭工程做得好，系统可靠性也会高

### 第二层：详细拆解

#### Harness的核心职责（6个维度）

**1. Context Engineering (上下文工程)**

What：什么信息要传给模型，什么不要传

Why：
- 令牌限制：上下文窗口有限，不能把所有信息都放进去
- 噪声管理：无关信息会影响性能
- 成本优化：更少的令牌 = 更低的成本

How：
- 精选相关信息（SQL查询结果 vs 整个数据库模式）
- 结构化组织（用XML标签清楚地标记不同部分）
- 去噪（过滤掉无关或矛盾的信息）

示例：
```
错：直接把整个数据库模式告诉模型（1000+个令牌）
对：只给出这个查询需要的表和字段（50个令牌）
```

**2. 工具抽象与编排**

What：把复杂的应用接口和数据库操作封装成模型能理解的接口

Why：安全性、可控性、一致性

How：
- 中间层包装：模型不直接调用数据库，而是调用一个安全的工具接口
- 错误处理：工具调用失败了怎么办
- 工具间依赖管理：某些工具需要其他工具的结果作为输入

示例：
```
错：模型直接写SQL："SELECT * FROM users WHERE id = ??"
对：模型调用工具："query_user_by_id(user_id=123)"
    → 工具内部验证 + SQL生成 + 错误处理
```

**3. 验证**

What：确保模型的输出正确，执行的行动安全且符合规则

Why：防止幻觉、防止危险操作、确保业务逻辑遵循

How：
- 输出验证：模型的输出格式是否正确、逻辑是否正确
- 业务规则检查：这个操作是否符合业务规则
- 权限检查：用户是否有权限做这个操作

示例：
```
模型决定："转账100万元"
验证会检查：
  - 账户余额 ≥ 100万
  - 用户权限允许转账
  - 转账金额 < 每日限额
  如果任何check失败，halt此操作并report error
```

**4. 错误恢复与重试**

What：工具调用失败了怎么处理

Why：提升系统可靠性

How：
- 重试逻辑：什么错误应该重试，什么应该快速失败
- 指数退避：重试的时间间隔怎么设置
- 断路器：某个工具一直失败，停止调用它
- 降级策略：工具不可用时，有没有替代方案

示例：
```
数据库连接超时
→ 可能是暂时性的，带指数退避重试
语法错误
→ 不会通过重试修复，立即失败
API速率限制
→ 等待然后重试
```

**5. 状态管理**

What：Agent执行过程中的状态怎么保存、中断后怎么续上，以及如何积累和学习经验

Why：
- 可恢复性：系统崩溃后能从检查点恢复
- 持久化：长期运行的任务需要保存进度
- 学习能力：系统能从历史经验中学习和优化

How：

**1. 执行状态**
- 检查点：定期保存执行状态
- 持久化存储：把状态存到数据库/磁盘
- 版本控制：追踪状态的变化
- 恢复：从检查点恢复

**2. 对话历史**
- 记忆过去交互：记住用户说过什么
- 为未来决策提供上下文：用历史指导现在
- 学习用户偏好：从对话中学习用户习惯

**3. 经验学习**
- 失败案例记录：记录失败的案例和原因
- 模式识别：观察什么条件下容易失败
- 决策优化：基于失败案例调整未来策略

这个机制类似于传统模型工程中的"经验教训反馈"：
- 识别失败模式（这类任务总是失败）
- 记录上下文（失败发生在什么条件下）
- 调整策略（下次遇到相似情况，用不同方法）

**示例：**
```
场景：Agent执行文本分析任务
历史记录：
  任务1（短文本）→ Haiku成功
  任务2（长文本）→ Haiku失败 → 升级到Sonnet成功
  任务3（复杂推理）→ Sonnet失败 → 用Opus成功
  ...

系统学习模式：
  "长+复杂的文本分析任务，不应该用便宜模型"
  
未来相似任务：直接路由到合适的模型，而不是重复便宜模型的失败
```

**关键观察**：状态管理不仅是"保存进度"，更是"系统的学习机制"。好的驾驭工程能从执行历史中不断优化自己的决策。

**6. 人工控制**

What：什么时候需要人工审批、怎么让用户干预

Why：治理、风险管理、可控性

How：
- 审批工作流：定义哪些操作需要审批
- 决策检查点：Agent规划完成后，展示给人类然后等待批准
- 覆盖机制：人类能覆盖Agent的决策
- 审计日志：记录所有决策和审批

示例：
```
转账操作
→ Agent计划：转账100万到账户X
→ 系统识别：这是高风险操作
→ 暂停，通知用户
→ 用户确认 → 执行
→ 审计日志：记录Agent计划、用户批准、执行结果
```

#### Agent vs 驾驭工程的关系

```
Agent（大语言模型核心）
    ↓
决策：要做什么、怎么做
    ↓
    ↓
驾驭工程（基础设施）
    ↓
确保：能可靠地做、能安全地做、能高效地做
```

**关键观察**：
- 如果Agent能力↑但驾驭工程不变 → 系统能做的事更多，但可靠性可能↓（更多复杂性）
- 如果Agent能力不变但驾驭工程↑ → 相同能力下可靠性大幅提升

所以不是"Agent强就够了"，而是"Agent和驾驭工程同样重要"。

#### 驾驭工程现在的问题

**问题1：规则驱动**

所有决策都是硬编码规则：
```
if error_type == "network_timeout":
    retry_count = 3
elif error_type == "validation_error":
    retry_count = 0
```

问题：这些规则是否最优？对不同的任务类型是否有不同的最优策略？

**问题2：无法学习**

系统不知道这些规则是否在工作：
- 某个任务类型：其实需要4次重试才能成功，但规则设定是3
- 某个任务类型：其实第一次失败就应该停止，但规则设定重试3次浪费时间

**问题3：工程复杂度高**

每个应用都要自己定义这些规则 → 代码重复、维护噩梦

### 第三层：Extensions

#### 驾驭工程的演进方向

**方向1：智能化（从规则驱动 → 数据驱动）**

```
现在：if condition then action（硬编码）
未来：模型学习根据历史数据采取什么行动

示例：
任务类型A：失败率15%，需要4次重试才能成功
任务类型B：失败率5%，需要1次重试才能成功
系统观察这个模式，自动学会对不同任务类型使用不同策略
```

How：
- 用元学习收集每个任务类型的执行数据
- 学习最优的重试策略、超时、模型选择
- 当新模式出现时持续适应

影响：从"基础设施"变成"优化引擎"

**方向2：自适应（从静态配置 → 动态优化）**

```
现在：timeout=30s，retry=3，use_model=Claude_Opus（固定）
未来：参数根据以下条件动态调整：
  - 当前任务复杂度
  - 系统负载
  - 成本预算
  - 用户的时间/成本偏好
```

示例：
```
下午3点，高系统负载
→ 超时自动↑（失败前等待更长时间）
→ 重试次数自动↓（减少重复计算）
→ 模型选择自动↓（使用Haiku而不是Opus以节省成本）

凌晨2点，低系统负载，用户不赶时间
→ 超时自动↑（可以等待更长时间）
→ 重试次数自动↑（确保成功）
→ 模型选择自动↑（使用Opus提升质量）
```

**方向3：托管化（从DIY框架 → 托管服务）**

```
现在：开发者使用LangGraph/LlamaIndex构建自己的驾驭工程
未来："Anthropic驾驭工程即服务"（类似AWS RDS）
  - 用户只需定义业务逻辑
  - 平台负责驾驭工程、扩展、可靠性
```

类比：
```
之前：自己管理数据库 → 复杂、高风险
现在：使用RDS → 只需定义模式和查询

之前：自己构建驾驭工程 → 复杂、高风险
未来：使用"Anthropic Agent Runtime" → 只需定义业务逻辑
```

影响：
- 降低构建Agent的门槛
- 让小团队也能构建可靠的Agent
- 驾驭工程变成商品化基础设施

#### 驾驭工程和基模升级的关系

一个关键问题：基模升级能否解决驾驭工程的问题？

**可靠性问题**
- 基模更强 → 单步成功率↑ → 整体成功率↑
- 但系统级问题（网络超时、应用接口故障、权限错误）无法通过模型强度解决
- 所以：可靠性问题部分能被吞掉，但基础设施级的容错无法消亡

**成本问题**
- 10步任务 = 10次调用，无论模型多强
- 基模升级无法改变这个基本事实
- 只能通过智能路由（什么时候用便宜模型、什么时候用强模型）来优化
- 所以：成本问题完全无法被基模吞掉

**可控性问题**
- 更强的模型反而可能更自主，更难控制
- 治理永远是治理，无法通过模型强度解决
- 所以：可控性问题无法被吞掉

**结论**：基模升级会改变驾驭工程的某些方面（比如错误恢复的重试次数），但不会使驾驭工程变得无关。

## VIII. 概念间的关系与融合

### 第一层：速览

这些概念不是独立的孤岛，而是在不同维度相互作用。理解它们之间的关系，能帮助你看清"什么会融合，什么会独立演进"。

### 第二层：关系地图

#### 关系1：Prompt Engineering ↔ Context Engineering ↔ Harness Engineering（主线演进）

**主线演进（功能需求驱动）：**

```
Prompt Engineering (2023-2024)
"让单次大语言模型调用更好"
    ↓
Context Engineering (2024-2025)
"让上下文管理更全面系统"
包括：上下文压缩、信息精选、结构化组织、动态组装
    ↓
Harness Engineering (2025+)
"让整个Agent系统可靠运行"
```

**平行线（支撑Harness Engineering）：**

**Skill System (2025下半年，Anthropic主推)**

价值主张：为Harness Engineering提供标准化、可复用、可评估的任务/工具模板

出现逻辑：
- Agent时代需要可靠的任务执行
- 需要标准化的执行单元 
- Skill被创造出来

关键优势：
- 可观察：每个Skill的执行都可被追踪
- 可复用：好的Skill可在不同Agent间共享
- 可评估：成功率、成本都可被测量
- 可优化：基于数据可迭代改进

**架构关系：**

```
Harness Engineering (顶层：让整个系统reliable)
  ├─ Context Engineering (信息准备层)
  ├─ Skill System (标准化执行单元)
  ├─ State Management (状态维护)
  ├─ Error Recovery (容错机制)
  └─ Verification & Validation (质量保证)
```

**关键区别**：Skill不是Prompt Engineering的自然演进，而是为了解决Agent时代的可靠性问题而被创造的支撑系统。两者的生命周期是独立的。

#### 关系2：检索增强生成 ↔ Tool Calling

都是给Agent外部信息的方式，但mechanism不同：

```
检索增强生成：从知识库retrieve静态信息
Tool Calling：call外部系统获取动态信息

Example：
问题："什么是transformer attention"
→ 检索增强生成：retrieve论文内容
问题："今天天气怎么样"
→ 工具调用：调用天气API

未来：驾驭工程会统一管理这两个，系统自动决定用哪个
```

#### 关系3：微调 ↔ 检索增强生成 ↔ 少样本学习

都是让模型知识增强到某个特定领域：

```
维度：成本 / 灵活性 / 持久性

微调：成本高 / 不灵活 / 永久
↓（不推荐，使用以下两个）

检索增强生成：成本中等 / 灵活 / 动态可更新
少样本学习：成本低 / 最灵活 / 仅本次会话

竞争：越来越多场景用检索增强生成替代微调
```

#### 关系4：Agent ↔ 驾驭工程（最关键）

```
Agent：思考者
驾驭工程：执行者

Agent说："我需要做X"
驾驭工程确保："X被可靠、安全、高效地完成"

Agent能力↑ vs 驾驭工程设计 → 两者都同样重要

老的思维：Agent强就够了
新的思维：Agent + 驾驭工程的联合优化
```

#### 关系5：数据层 ↔ 驾驭工程（都critical，但维度不同）

```
数据层：决定"给模型什么信息"
驾驭工程：决定"模型怎么可靠地执行"

都不会被基模吞掉，但理由不同：
- 数据层：因为问题定义是永久的
- 驾驭工程：因为基础设施是永久的

两层都是系统质量的决定性因素
```

## IX. 市场现状与竞争格局

### 第一层：速览

2026年的AI Agent和驾驭工程市场处于一个关键的市场整合阶段。框架大战已经基本结束，现在的竞争转向"谁的驾驭工程更好"。

关键事实：
- Agent框架已经有明确的市场赢家
- 基础设施提供商（Anthropic、OpenAI）正在优先发展自己的产品
- 开源生态面临新的经济压力

### 第二层：详细分析

#### Agent Framework的现状

根据最新的industry analysis：

**LangGraph赢得了有状态多步骤工作流的市场**

LangGraph的图形化执行模型能清晰地映射到复杂业务逻辑的心智模型上。如果你的Agent需要暂停、等待人工审批、从检查点恢复或在中间输出处分支，LangGraph是最自然的选择。

特点：
- 显式状态管理：整个执行历史都是可检查的
- 支持条件边：根据当前状态动态决定下一个节点
- 支持循环：与传统有向无环图(DAG)不同，LangGraph被设计用来处理循环

权衡：
- 代码相对冗长：需要定义状态模式、节点函数、边的条件
- 但对于生产规模的工作流，这种冗长是值得的

**AutoGen赢得了多Agent协作的市场**

微软的AutoGen赢得了多Agent协作的细分市场——即需要多个专业Agent相互辩论、评论或并行工作的场景。

应用场景：
- 需要多个专业化Agent分工的系统
- Agent之间需要辩论或评价的工作流
- 并行处理能提升效率的场景

**CrewAI占据了易用性与功能的平衡点**

CrewAI占据了易用但强大的中间地带，采用基于角色的抽象，非机器学习工程师也能理解和部署。

特点：
- 用人性化的角色和任务来定义Agent
- 不需要深入理解图或状态机
- 团队成员不需要都是AI/ML工程师

限制：
- 当需求超出顺序/层级任务执行时，会遇到能力天花板
- 6-12个月后，很多团队反馈遇到这个限制，需要迁移到LangGraph

**LlamaIndex的角色转变**

LlamaIndex没有消亡，而是演进了。它现在是Agent系统内检索层的主流选择，不再是编排层。

新的模式：
```
LlamaIndex：构建知识流水线和检索层
        ↓
LangGraph/AutoGen：编排查询这些检索结果的Agent

这种职责划分已经成为稳定的模式
```

LlamaIndex工作流：
- 事件驱动的编程模型用于Agent执行
- 特别适合检索增强生成(检索增强生成)重型Agent
- 但不适合通用编排（缺乏处理外部应用接口调用的好原语）

**微软Agent框架正式发布**

微软在2025年10月合并了AutoGen和Semantic Kernel成统一框架。正式发布预计在2026年Q1，包含生产级SLA、多语言支持（C#、Python、Java）和深度Azure集成。

战略意义：
- 微软正式押注Agent作为企业未来
- 为.NET企业和被Azure锁定的公司提供原生选项

**市场现状小结：框架大战基本结束**

2023-2024年框架爆炸期已经大幅收窄。赢家已经确定。

现在的选择：
- 有状态、可控的工作流 → LangGraph
- 多Agent协作 → AutoGen / 微软Agent框架
- 简单、易用的工作流 → CrewAI
- 检索优先 → LlamaIndex工作流
- 定制化需求 → 自己搭建或组合多个框架

#### Anthropic的策略转变：OpenClaw禁用案例

这个案例很好地说明了AI基础设施提供商正在发生的转变。

**事件时间线**

从2026年4月4日下午12点(PT)开始，Anthropic阻止了Claude Pro和Max订阅用户使用订阅来支撑OpenClaw这样的第三方Agent工具。

背景：
- OpenClaw创建者Peter Steinberger在2月14日加入了OpenAI，OpenClaw被移交给开源基金会
- OpenClaw估计有135,000+个运行实例

**官方理由**

Anthropic声称第三方工具对其系统造成了"过度负担"，迫使其优先为自有产品和API用户提供服务。

技术根源：
- OpenClaw这样的第三方驾驭框架绕过了提示词缓存优化
- 一个重度使用的OpenClaw会话消耗的基础设施资源远多于等效的Claude Code会话
- 第三方工具实际上以固定订阅价格获得了无限计算资源

**开发者反应**

社区估计约60%的活跃OpenClaw会话是通过订阅配额运行的。

对用户的影响：
- 从固定价格订阅 → 按令牌计费
- 成本可能增加50倍（取决于使用模式）
- 很多业余开发者无法承受继续使用

**战略意义**

这不是简单的"Anthropic要消灭竞争对手"，而是：

1. **经济现实检查**
   - 订阅模式无法支撑24小时Agent使用
   - OpenClaw无意中成为了Anthropic最大的补贴接收者
   - 这种模式无法持续

2. **产品优先级**
   - Anthropic希望更好地服务自己的产品（Claude Code、Claude Projects）
   - 第三方工具消耗的容量代表了机会成本

3. **更广泛的行业模式**
   - Google最近也对连接第三方工具的Gemini CLI用户采取了行动
   - 当Agent工作负载成本变得清晰后，最初受益于第三方生态驱动的AI公司现在都在限制访问
   - 这不是Anthropic独有的做法，而是所有提供商都在进行的

**缓冲措施**

Anthropic提供了：
- 等于用户月度订阅费用的一次性配额，可在4月17日前使用，加上预购额外使用包最高30%的折扣

开源贡献：
- Anthropic执行人Boris Cherny亲自提交了改进提示词缓存效率的代码，专门为OpenClaw优化

这个举动表明Anthropic仍然支持开源，但前提是不能以牺牲自身基础设施经济性为代价。

#### 未来的可能格局

根据这些信号，市场可能演进成：

```
Anthropic：
  - 第1层：直接产品（Claude.ai、Claude Code、Claude Projects、Cowork）
  - 第2层：API用户（直接使用API的企业）
  - 无第3层：第三方驾驭框架使用固定订阅（已明确关闭）

OpenAI：
  - 预期采用类似策略（Steinberger加入OpenAI后）
  - 但可能对第三方生态的支持更积极
  
第三方开发者：
  - 必须采用按量计费或自己管理基础设施
  - 无法依靠API补贴来构建产品

开源：
  - 继续存在，但经济学发生变化
  - 用户基数较小（仅限有运营预算的用户）
  - 更少依赖单一提供商
```
