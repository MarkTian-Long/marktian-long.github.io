# OpenClaw 爆红背后：一个品类的诞生与宿命

> 一个奥地利独立开发者用一个"危险"的开源项目验证了 personal agent 的市场需求，然后被 OpenAI 招走了。这件事本身，比任何关于 Agent 未来的预测都更值得认真拆解。

我第一次听说 OpenClaw 是去年年底，当时没有用，配置成本摆在那里，token 也要自己掏钱，动力不足。真正开始用，已经是今年二三月份，国内大厂陆续跟进，部署门槛低了，还送 token。

这个时间差本身就很说明问题。一个产品爆红之后，真正推动普通技术用户开始用的，不是它本身，而是跟进者。OpenClaw 验证了需求，别人降低了门槛。这件事的逻辑，贯穿了这款产品的整个故事。

## 它到底是什么

很多人第一次看到 OpenClaw 的介绍，会以为它是一个"更强的 AI 助手"。这个理解是偏的。

OpenClaw 不是模型，也不是产品意义上的 AI 助手。[它是一个 agentic harness](https://fortune.com/2026/03/14/openclaw-china-ai-agent-boom-open-source-lobster-craze-minimax-qwen/)，一套把 LLM 包装成能自主执行任务的基础设施框架。你选一个模型作为大脑（Claude、GPT-4、DeepSeek 都行），OpenClaw 负责把这个大脑接上你的工具：邮件、日历、文件系统、浏览器、各种 API。然后你通过 Telegram 或 WhatsApp 给它下指令，它去执行。

更准确的类比是：OpenClaw 是脚手架，不是房子。

在它出现之前，一个开发者如果想让 AI 帮他持续、自动地跨 App 执行任务，选项是从零搭：自己写调度逻辑、自己维护工具调用、自己处理记忆和状态管理。OpenClaw 把这些通用底层打包好了，用户只需要在上面叠自己的 skill。

这个定位决定了它的真实用户画像，不是所有人想象中的"普通大众"，而是**有明确自动化需求、愿意付出配置成本换自动化收益的跨行业技术爱好者和个体创业者**。[实际社群数据印证了这一点](https://36kr.com/p/3709855637598595)：用户里有技术从业者，也有行政、律师、财务、淘宝店主，年龄从05后大学生到中年企业高管都有，共同点不是职业，而是"有高频重复任务想甩给 AI"这个具体需求。

但这里有一个值得正视的反差：[有知乎作者统计，90% 的人装完 OpenClaw 之后，根本不知道该拿它干什么](https://zhuanlan.zhihu.com/p/2012431646817813094)。爆红和真实有效使用之间，存在一道不小的鸿沟。这道鸿沟的本质是：OpenClaw 给你最大的自由，但也把最大的责任推给了你，你得先想清楚要自动化什么，工具才有意义。

## 三个产品决策，每一个都有代价

[OpenClaw 能在 60 天内拿到 25 万 GitHub stars](https://en.wikipedia.org/wiki/OpenClaw)，不是偶然。它的三个核心产品决策叠在一起，产生了引爆效果。但每一个决策，背后都有明确的 trade-off。

**第一个决策：用 IM 做界面。**

Steinberger 没有做独立 App，而是让 OpenClaw 通过 Telegram、WhatsApp、Discord 接受指令。表面上看是"降低用户适应成本"，但更深的逻辑是两层：

一是对用户，IM 已经在他们的注意力流里了。你不需要主动切换到一个新 App 去"使用 AI"，AI 就在你每天本来就会打开的地方。这不只是熟悉感，是零切换成本。

二是对开发者，用 IM 做通道意味着服务器、推送、在线状态维持这些基础设施全部不用管，这些成本转嫁给了 Telegram 和用户自己。一个独立开发者，省掉这些，才能把精力放在核心逻辑上。

代价是：IM 界面天然缺乏权限管控层。你和 AI 的交互通道，就是你日常聊天的通道，边界模糊。

**第二个决策：开源。**

这个决策很多人会理解为"放弃商业价值"。我觉得恰恰相反，开源是他在个人开发者身份下能选择的最优路径。

闭源产品的护城河靠的是功能积累和用户数据。一个人跑不过团队，团队跑不过社区。开源之后，[中国开发者自发适配了 DeepSeek 和微信，腾讯和 Z.ai 宣布了基于 OpenClaw 的服务](https://en.wikipedia.org/wiki/OpenClaw)，ClawHub 上出现了上万个 skill，这些都是 Steinberger 一个人永远做不到的。

开源换来的不是钱，是**生态扩张速度**和**个人 credential**。项目爆红后不久，他加入了 OpenAI。

**第三个决策：全权限。**

OpenClaw 能读邮件、操作文件、跨 App 调用，权限范围极大。这是它能力上限的来源，也是它最大的安全隐患。[Cisco 的 AI 安全团队测试了一个第三方 OpenClaw skill，发现它在用户无感知的情况下完成了数据窃取和提示词注入](https://en.wikipedia.org/wiki/OpenClaw)。[ClawHub 上一度有 12% 的 skill 包含恶意代码](https://zhuanlan.zhihu.com/p/2012431646817813094)，这不是小概率事件。

但这里有一个值得想清楚的问题：全权限是 bug 还是 feature？

如果把 OpenClaw 的权限缩小，不能读邮件、不能操作文件，它还是"贾维斯"吗？不是了，它变成了一个普通聊天机器人。能力边界和安全边界在这个产品上天然冲突，没有办法两全。

Steinberger 的选择是：先用全权限把能力上限拉满，代价后置。用一个"危险"的版本引爆市场，然后靠社区和生态去补安全。这是一个清醒的策略选择，不是疏忽。

## OpenClaw 走进 B 端的真实路径

OpenClaw 爆红之后，有一种声音说它会改变企业的工作方式。这个方向是对的，但路径值得仔细看。

OpenClaw 原生版本直接走不进 B 端。[企业级部署需要满足高安全、高稳定、高兼容、可管控四大核心需求，而 OpenClaw 原生部署在多终端兼容性、权限管控、审计链路上存在明显短板](https://www.53ai.com/news/Openclaw/2026030986915.html)。这些不是"模型更强"就能解决的，也不是一个开源脚手架的设计目标。

真正让 OpenClaw 触达 B 端的路径，是大厂的二次封装。[阿里云、腾讯云、中兴通讯、中关村科金几乎同步推出了各自的企业级 Claw 解决方案](https://finance.sina.com.cn/tech/roll/2026-03-10/doc-inhqpaep9999746.shtml)，核心都是在原生 OpenClaw 之上加了一层：私有化部署、数据不出域、多租户隔离、审计链路。飞书和企业微信也相继打通了 OpenClaw 接入，让员工可以通过 IM 直接处理文档、日程等任务。

这个现象本身就值得认真看：OpenClaw 定义了品类，但真正落地 B 端的，是在它基础上做了一层产品化封装的方案。脚手架走不进企业，企业需要的是有人帮它把脚手架装好、调好、保好。

B 端的机会不在 OpenClaw 本身，而在"谁能把 OpenClaw 的能力包装成企业真正可用的形态"。目前来看，这个位置正在被云厂商快速占据。

## 品类宿命：脚手架终将被基础设施吞并

这是我觉得关于 OpenClaw 最值得想清楚的一个问题。

它证明了一件事：**用户对 personal agent 的需求是真实的，市场空白是真实的**。但证明市场空白存在，和能长期占据这个市场，是两件事。

脚手架类产品有一个结构性的天花板：一旦更底层的基础设施开始原生支持同类能力，脚手架的价值就会被快速压缩。

从目前的趋势来看，最可能的打法是 OS 层渗透，agent 不住在某个 App 里，而是住在操作系统里。Apple Intelligence 是这个方向的早期尝试，国内厂商也在沿着自己的软件生态做整合。OS 层的 agent 有天然优势：权限由系统管理、不需要用户手动配置、token 成本由平台承担。

OpenClaw 现在的核心优势，开源、可定制、全权限，在 OS 层整合之后反而可能变成劣势。普通用户不需要这种控制粒度，他们需要的是"直接能用、出了问题有人负责"。

这不是说 OpenClaw 会消失。它大概率会在开发者工具的细分赛道里存活，继续作为极客和技术用户的首选框架。但成为主流用户的 personal agent 入口，这条路大概率会被 OS 层的原生能力截断。

值得一提的是，Steinberger 在项目移交给非营利基金会之后选择加入 OpenAI，去更底层的地方参与下一代 agent 基础设施的建设。这个选择是否出于对 OpenClaw 长期前景的判断，外人无从得知。但从结果上看，对一个个人开发者来说，这或许是这个故事里最合理的一种结局。

---

## 最后说一句

OpenClaw 的爆红，很容易被解读为"agent 时代到来了"。我倒觉得，它更准确地揭示的是：**agent 的基础设施还远没有成熟，但需求已经真实存在了**。

市场在等一个门槛更低、更安全、不需要用户自己当系统管理员的版本。OpenClaw 是这个版本出现之前的过渡形态，它证明了方向，但自己可能不是终点。

至于全自动、无监督的 agent 什么时候能真正大规模落地，我觉得 2027 年之前可能性不大。C 端的信任壁垒不会因为入口变了就消失，B 端的权责问题也不会因为模型更强而自动解决。但随着使用习惯的建立，这些壁垒会慢慢松动。

方向是对的，时间表还需要等。
