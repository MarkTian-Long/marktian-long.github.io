# ASCI 科研任务执行系统 · 产品设计文档

**作者**：刘洋 Liu Yang  
**岗位**：AI 产品经理（ASCI）· 深圳赛因智科科技  
**作业题**：请设计一个「科研任务执行系统」

---

## 一、公司与赛道背景理解

### 1.1 赛因智科 / OPENSCI 是什么

赛因智科是 OPENSCI 项目的国内执行主体。OPENSCI 由**世界顶尖科学家协会（WLA，World Laureates Association）**发起，于 2026 年 2 月在迪拜 WLA 峰会上正式对外宣布，顾问网络覆盖 78 位诺贝尔奖得主。实际操盘人 Michael Gan 是 KuCoin 联合创始人，现任 WLA 副秘书长——这意味着项目同时拥有顶级学术背书和加密/Web3 资本背景。

公司做两条产品线，但本质上是一体两面：

- **OPENSCI**：科研价值分配基础设施，解决"谁创造了这个发现、应该获得多少回报"的问题。通过链上身份（IDA）、自动分配协议（GRSA）、Publish-to-Earn 模型，把科研贡献从学术头衔变成可量化的经济回报。公司将其命名的核心问题叫 **"Goodenough Gap"**——John Goodenough 用一生研究锂电池正极材料，100 万美元的研究投入催生了 2.5 万亿美元产业，但价值从未回流到原始创造者。

- **ASCI（Artificial Science Intelligence）**：科研 AI OS，负责科研行为的生产力层。文献检索、知识整理、假设生成、实验设计、数据分析、论文写作全链路 Agent 能力。ASCI 产生的每一步科研行为数据，都会通过 OPENSCI 的链上系统转化为可溯源的价值记录。

**一句话总结**：OPENSCI 解决科研价值的分配问题，ASCI 解决科研效率的生产问题，两者合力构建"科研价值工厂"。

---

### 1.2 为什么现在是这个赛道的窗口期

AI for Science 作为叙事在 2024-2026 年集中爆发，有几个结构性原因：

**算力侧**：NVIDIA GTC 2026 将科研加速作为核心主题——Blackwell 架构将蛋白质结构预测加速超 100 倍，Earth-2 用 AI 物理模型重建全球气候系统，BioNeMo 处理 170 万个高置信度蛋白质复合物。算力基础设施已经就位。

**工具侧**：目前市场上的科研 AI 工具普遍是**单点工具**，没有形成完整的工作流闭环：

| 产品 | 定位 | 局限 |
|---|---|---|
| Elicit | 文献检索 + 数据提取 | 单点工具，无 Agent 编排 |
| Consensus | 基于论文的问答 | 覆盖面有限，无完整工作流 |
| ResearchRabbit | 引用网络可视化 | 发现工具，不解决生产问题 |
| Perplexity Academic | 实时学术搜索 | 泛用，缺乏科研场景深度 |
| NotebookLM | 私有文档问答 | 无多 Agent、无协作能力 |

**ASCI 要填的空缺**：覆盖科研全链路的 Agent OS，而不是某个环节的辅助工具。这个位置目前没有成熟竞品占据。

**需求侧**：科研人员承受的信息过载越来越严重——每年超过 400 万篇论文发表，顶级期刊接受率持续下降，大量有价值的基础研究因为"找不到相关工作"或"无力完成系统综述"而搁置。

---

### 1.3 ASCI 产品的核心挑战

做科研场景的 Agent OS，和做普通效率工具有本质区别：

**科研任务的特殊性**：
- 任务链路长，中间步骤强依赖前序输出质量
- 用户（科研人员）对错误极度敏感——一个错误引用可能毁掉一篇论文
- 许多判断没有客观标准，涉及学科领域的隐性知识
- 结果的可解释性要求远高于一般商业产品

**这直接决定了 ASCI 的产品设计核心命题**：如何在 AI 执行效率和人类可信任度之间找到可操作的边界。

这也是本次作业题想考察的本质问题。

## 二、作业题解读

### 2.1 题目约束条件

题目明确了产品需要实现的**四个用户目标**：

1. **提交任务** — 用户能够清晰表达意图并启动执行
2. **理解执行过程** — 过程不能是黑盒，用户需要知道 AI 在做什么
3. **判断结果是否可信** — 用户需要有依据评估输出质量，而不是盲目接受
4. **在必要时进行干预或修正** — 系统必须为人工介入留出通道

这四个目标本身就是产品的功能骨架，对应 Demo 的四个核心模块设计。

题目同时给出了**四个系统约束**，这是设计难点的来源：

- **任务复杂（非单步）** → 需要任务拆解和多层 Agent 编排，不能把科研任务当成一次 LLM 调用处理
- **AI 可能出错（推理错误 / 数据错误）** → 错误是必然发生的，产品设计必须预设错误处理路径，而不是假设 AI 总是正确
- **执行过程不可完全黑盒** → 透明度是基本要求，不是附加功能
- **用户不一定是 AI 专家** → 技术细节需要被翻译成科研语言，置信度、推理链路、降级策略都要用用户能理解的方式呈现

### 2.2 四道面试题的考察本质

四道面试问题实际上在考察同一件事的四个切面：

| 面试题 | 考察点 |
|---|---|
| 系统最容易出错的环节在哪里？ | 你是否真正理解 AI 的能力边界 |
| Agent 连续3次出错你怎么处理？ | 你是否设计了降级策略和反馈机制 |
| 用户为什么会相信这个结果？ | 你是否考虑了结果的可解释性与信任建立 |
| 哪一步必须由人来做？ | 你是否清楚 AI 的不可替代边界在哪里 |

四道题的底层是同一个设计命题：**在科研场景下，如何定义 AI 的边界，让人机协作产生比两者单独工作更好的结果。**

## 三、产品设计思路

### 3.1 整体框架：四层设计

```
用户层    ─  提交任务 / 理解过程 / 判断结果 / 干预修正
             ↕
透明层    ─  让用户在任意时刻知道"AI 在做什么、为什么这样做、有多确信"
             ↕
执行层    ─  多层 Agent 任务拆解与执行，含风险标注和置信度评估
             ↕
信任层    ─  结果溯源、可信度量化、人工介入记录
```

核心设计原则：**过程透明优先于结果完美**。科研场景里，用户不需要 AI 永远正确，但需要随时知道 AI 在哪里可能出错、出错了怎么办。

### 3.1.1 Demo 需要体现的 Agent 能力维度

根据岗位职责对「Agent 能力产品化」的定义，Demo 中需要可见地体现以下8个维度：

| 能力维度 | 在 Demo 中的体现位置 |
|---|---|
| 任务拆解 | Screen 2 左列：多层任务树，根节点拆解为5步+子步骤 |
| 工具调用 | 执行日志中标注每步调用的工具（如 PubMed API、统计分析模块） |
| 工作流编排 | 任务树的执行顺序与依赖关系，子任务完成后触发父任务状态更新 |
| 记忆机制 | 日志中体现前序步骤输出如何传递给后续步骤（上下文累积） |
| 知识库 | 文献库检索步骤，显示数据来源（PubMed / arXiv） |
| 上下文管理 | 连续出错时「扩大上下文窗口」作为第一条重试策略 |
| 人机协同 | Human Checkpoint 节点 + 异常处理时的人工接管路径 |
| 结果可解释性 | Screen 3 可信度拆解 + 来源溯源，每条结论可回溯来源 |

---

### 3.2 回答面试题 Q1：最容易出错的环节

科研 Agent 链路中，风险不是均匀分布的，从低到高大致分三级：

**低风险**（规则明确，AI 稳定）：
- 关键词提取
- 数据库检索
- 格式化输出

**中风险**（需要判断，AI 可能漂移）：
- 摘要相关性过滤（无法识别领域隐性知识）
- 数据统计方法选择（分布异常时模型可能默认错误处理方式）
- 综述报告生成（结构和重点存在主观判断）

**高风险**（推理密集，最易出错）：
- **全文精读与摘要**：易出现幻觉引用、断章取义
- **假设生成**：因果推理链路长，结论容易与已有研究矛盾
- **跨文献结论综合**：信息来源多，一致性校验困难

**设计回应**：任务树中每个节点标注风险等级，高风险节点在执行前提示用户、执行中加强置信度监控、执行后要求确认再推进。

---

### 3.3 回答面试题 Q4：哪步必须由人来做

**不能交给 AI 的判断**，本质上是"需要领域隐性知识"或"涉及价值判断"的步骤：

1. **研究边界的确认**（关键词选取之后）：什么在研究范围内、什么不在，这是研究问题的定义，AI 无法替代研究者做这个决策
2. **文献入选标准**（摘要筛选之后）：哪些论文算"高质量相关"，涉及领域认知，AI 的过滤规则无法覆盖隐性知识
3. **假设的科学合理性判断**（假设生成之后）：AI 生成的假设是否值得深入，需要研究者用领域经验判断

**设计回应**：这三类节点在任务树中标注 🔒 Human Checkpoint，执行完成后系统自动暂停，等待用户确认再进入下一阶段。这不是产品缺陷，而是一个主动的设计声明：ASCI 知道自己的边界在哪里。

---

### 3.4 回答面试题 Q2：连续出错的处理策略

三次出错不是一个简单的重试问题，背后是三种不同的失败模式：

**第一次出错**：可能是随机噪声，重试即可，同时扩大上下文窗口
**第二次出错**：可能是系统性偏差，换参数（降低温度）或换模型
**第三次出错**：模型能力边界问题，继续重试无意义，必须人工介入

**降级策略的三条路径**：

| 策略 | 适用场景 | AI 角色 |
|---|---|---|
| 调整参数重试 | 推理链路断裂，随机性问题 | 主导执行 |
| 切换备用模型 | 输出与已有文献矛盾，事实性错误 | 主导执行，换引擎 |
| 人工接管 | 置信度极低，科学判断超出 AI 能力 | 辅助，提供草稿，人做决策 |

**设计回应**：第三次报错后，执行日志出现显著红色警告，右侧状态面板切换为「异常处理模式」，展示失败原因分析和三条降级路径。用户选择后系统继续执行，如果选择人工接管，AI 提供草稿供用户编辑，编辑完成后再接回自动流程。

---

### 3.5 回答面试题 Q3：用户为什么相信结果

信任不是靠宣称"AI 很准"建立的，而是通过**让用户看到判断依据**来建立的。

可信度需要拆解成可观测的维度。参考岗位职责对「AI 产品评估机制」的定义（效果、稳定性、成本、时延、安全性），结合科研场景实际，落地为以下五个维度：

| 维度 | 科研场景含义 | 对应 JD 评估维度 |
|---|---|---|
| 来源质量 | 引用文献的期刊影响因子、文献数量、发表年份分布 | 效果 |
| 推理链路 | 每个结论可回溯证据，无跳跃性推断 | 效果 |
| 数据一致性 | 跨步骤结论无矛盾 | 稳定性 |
| 执行稳定性 | 该任务类型的历史成功率、是否触发过重试 | 稳定性 |
| 安全边界 | 是否涉及未经证实的医学/科学声明，高风险结论是否有警示 | 安全性 |

> Demo 中展示前三个维度（来源质量 / 推理链路 / 数据一致性）作为核心评分，后两个作为辅助提示出现，避免界面过于复杂。

总分低或某个维度异常时，系统不是隐藏问题，而是主动给出建议（如：「建议在引用前对摘要筛选步骤进行人工复核」）。

更重要的是：全程可溯源。用户在结果页能看到每个结论对应的原始来源，每个步骤的执行记录，以及哪些步骤经过了人工干预——这种透明度本身就是信任的来源。

## 四、产品结构概览

### Screen 1：任务提交
- 任务输入框 + 3个预设场景快捷选择（覆盖低/中/高风险，分别对应正常完成、触发警告、触发人工介入三种演示路径）
- 预设场景：文献综述 / 数据分析 / 研究假设生成

### Screen 2：执行过程（核心页）
三列布局：
- **左：任务拆解树** — 多层 Agent 结构，含风险标注、Human Checkpoint 标记、节点实时状态
- **中：执行日志流** — 终端风格，实时追加，含 INFO / WARN / ERROR 分级，连续出错时显示红色警告条
- **右：状态面板** — 正常时显示当前步骤置信度 + 进度；异常时切换为降级策略选择面板

底部控制栏：自动推进 + 随时可暂停 / 强制介入

### Screen 3：结果与评估
- 结果内容 + 来源溯源列表
- 可信度三维度评分 + 总分 + 建议提示
- 操作：接受 / 请求修正 / 选择步骤重新执行
