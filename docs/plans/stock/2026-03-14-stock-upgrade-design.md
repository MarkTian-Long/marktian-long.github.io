# A股 AI 研究助手 — 升级设计方案

**日期**：2026-03-14
**作者**：刘洋
**状态**：已确认，待实施

---

## 一、背景与目标

### 现状问题
当前 A股助手是一个聊天框 + Yahoo Finance 数据的简单 pipeline，与通用大模型对话体验无本质差异，无法体现 AI 产品经理的专业能力。

### 升级目标
将工具重新定位为**「AI 产品能力展示台」**，每个功能模块对应一种核心 AI 产品设计能力，合在一起构成一个完整的金融 AI 研究工作台。

### 目标受众
产品/技术面试官。核心印象：**"这个 PM 懂得如何在正确的场景用正确的方式使用 AI"**。

---

## 二、设计原则

### 1. AI 能力显性化
每个模块标注所用 AI 技术（RAG / Agent / Structured Output / Prompt Engineering），让面试官一眼看到产品设计意图。

### 2. 过程透明化
AI 的"思考过程"尽量可见，不做黑盒。检索了哪些数据、用了什么 prompt、经过几步推理——都要展示。

### 3. 不确定性诚实表达
所有 AI 生成内容带置信度标注或"仅供参考"标记，体现对 AI 能力边界的产品认知。

### 4. 金融合规意识（贯穿全局 + 专项展示）
- **全局层面**：每个 Tab 有统一的免责声明、数据来源标注、AI 内容标记
- **专项展示**：「合规 & 数据血缘」Tab 显性化金融 AI 产品的合规设计思考

---

## 三、功能架构（5 Tab）

| # | Tab 名称 | 核心 AI 能力 | 数据状态 |
|---|---------|------------|---------|
| 1 | 行情助手 | Prompt 工程 + AI 工作流 | 真实（Yahoo Finance） |
| 2 | 个股诊断 | 结构化输出 + 模型选型 | 混合（行情真实 + 基本面 mock） |
| 3 | 研报生成 | RAG 设计 | Mock（检索过程可视化） |
| 4 | 市场雷达 | 多源数据整合 + 主动推送 | Mock（信息看板） |
| 5 | Agent 实验室 | AI Agent 设计 | Mock（执行过程可视化） |
| 6 | 合规 & 数据血缘 | 金融 AI 合规设计 | 静态展示 |

## 四、各模块详细设计

### Tab 1：行情助手（保留 + 增强）

**保留**：现有聊天交互 + Yahoo Finance 数据获取

**新增**：
- 每次 AI 回复旁显示可折叠的「Prompt 设计卡」：
  - 意图识别 prompt 片段
  - 模型选择理由（为什么用 stepfun/step-3.5-flash）
  - 该步骤的置信度
- 三步流程可视化保留，但增加每步的"设计说明"气泡

**体现能力**：PM 懂 prompt 设计，知道 LLM 在哪个环节用、怎么用、为什么这样用。

---

### Tab 2：个股诊断报告

**交互**：输入股票名 → 生成结构化「诊断卡」

**诊断卡结构**：
```
┌─────────────────────────────────────────────┐
│  贵州茅台 (600519.SS)         AI 综合评级     │
│  ████████████████             持有 · 置信度 72% │
├──────────┬──────────┬────────────────────────┤
│  技术面   │  基本面   │       情绪面            │
│  (真实)   │  (mock)  │       (mock)            │
│  近5日走势 │ PE: 28.3 │  新闻情绪: +3.2/5      │
│  ▲ +2.3%  │ 市值:2.1T│  ████████░░ 偏多       │
└──────────┴──────────┴────────────────────────┘
│ AI 解读（100字，带"仅供参考"标注）              │
└─────────────────────────────────────────────┘
```

**右上角「模型选型说明」**：
> 此模块使用结构化输出模式（JSON Schema 约束），而非自由对话，原因：金融数据需要字段确定性，避免 LLM 自由发挥导致格式不稳定。

**体现能力**：结构化输出设计、AI 置信度表达、模型选型思维、PM 对 AI 能力边界的理解。

---

### Tab 3：研报生成（RAG Demo）

**交互**：双栏布局

```
左栏：知识源面板              右栏：生成报告
┌──────────────────┐         ┌──────────────────┐
│ 检索知识库        │         │  [股票/行业输入框] │
│ 新浪财经 35万条   │         │                  │
│ ──────────────── │  ─────▶ │  正在检索...      │
│ 相关新闻 Top 3:  │         │  ──────────────  │
│ [1] 茅台Q3财报... │         │  研究报告正文      │
│ [2] 白酒行业...  │         │  （LLM 生成）     │
│ [3] 消费板块...  │         │                  │
│ 相关度: 0.87     │         │  数据来源: [1][2] │
└──────────────────┘         └──────────────────┘
```

**关键设计**：
- 检索过程可视化（显示"正在向量检索"动画）
- 每条检索结果显示相关度分数
- 生成报告中的引用来源可追溯回左栏新闻
- 底部说明：「为什么用 RAG 而不是直接让 LLM 回答」

**数据说明**：新闻数据为 mock，但检索 + 生成流程真实运行（基于 mock 数据）。

**体现能力**：RAG 系统设计、知识源管理、检索结果透明化、引用可溯源。

│  热点板块热力图   │  今日 AI 简报                  │

### Tab 5：Agent 实验室

**交互**：输入复杂问题 → 可视化 Agent 执行过程

**问题示例**："茅台现在值得买入吗？"

**执行过程可视化**：
```
用户问题
    │
    ▼
┌─────────────────────────────────────────────┐
│  🤔 THINK   分析问题需要哪些信息...           │
│             需要：价格数据、估值、市场情绪      │
├─────────────────────────────────────────────┤
│  📋 PLAN    制定执行计划                      │
│             Step 1: 查询近期行情              │
│             Step 2: 获取 PE/PB 估值          │
│             Step 3: 分析新闻情绪              │
│             Step 4: 综合判断                  │
├─────────────────────────────────────────────┤
│  ⚡ ACT     执行工具调用                       │
│             ✅ get_price("600519.SS", "5d") │
│             ✅ get_valuation("茅台") [mock] │
│             ✅ search_news("茅台") [mock]   │
├─────────────────────────────────────────────┤
│  👁 OBSERVE  整合结果，进行推理...             │
└─────────────────────────────────────────────┘
    │
    ▼
最终结论（带推理链，带"仅供参考"标注）
```

**工具定义展示**：右侧可展开「工具设计文档」，展示每个工具的 function calling schema。

**体现能力**：Agent 设计思维、工具调用设计、ReAct 范式理解、推理链透明化。

### Tab 6：合规 & 数据血缘

**定位**：显性化金融 AI 产品的合规设计思考，体现行业认知深度。

**内容结构**：

**Part A：金融 AI 合规设计原则**（静态展示）
- 为什么金融 AI 必须"可解释"：监管要求（证监会、银保监）+ 用户保护
- 本工具的合规设计：内容标注、不确定性表达、数据来源透明
- 生产环境需要增加的合规层：用户资质验证、投资建议豁免声明、操作日志

**Part B：数据血缘图**（可视化）
```
Yahoo Finance API
    │ (行情数据，15分钟延迟)
    ▼
数据清洗层（去重、异常值过滤）
    │
    ├──▶ Tab1 行情助手（原始数据展示）
    ├──▶ Tab2 个股诊断（技术面分析）
    └──▶ Tab5 Agent（工具调用）

新浪财经新闻（Mock）
    │ (35万条，2025年)
    ▼
向量化层（text2vec-base-chinese）
    │
    └──▶ Tab3 研报生成（RAG 检索）

模拟数据（标注 [MOCK]）
    └──▶ Tab2 基本面、Tab4 市场雷达
```

**Part C：AI 生成内容标准**
- 本工具所有 AI 输出的标注规范
- 置信度计算说明
- 免责声明全文

**体现能力**：金融行业 AI 合规意识、数据治理思维、产品设计的完整性（不只做功能，还考虑合规边界）。

## 五、全局合规设计

### 统一免责栏（每个 Tab 底部固定）
```
⚠️ 本工具为 AI 产品能力演示，所有内容仅供参考，不构成投资建议。
   带 [MOCK] 标注的数据为模拟数据。真实行情来源：Yahoo Finance（约15分钟延迟）。
```

### AI 内容标注规范
- 🤖 标签：标注 AI 生成内容
- 📊 标签：标注真实数据
- 🎭 标签：标注模拟数据
- 置信度：以百分比 + 色条展示（绿色 >70%，黄色 40-70%，红色 <40%）

### 数据源透明化
每条信息标注：`来源 | 获取时间 | 数据类型`

## 六、技术实现策略

### 真实 API（Tab 1、2 技术面、Tab 5 行情工具）
- Yahoo Finance（现有，保留）
- OpenRouter（现有，保留）

### Mock 数据设计原则
- Mock 数据要"看起来真实"（合理的数值范围）
- 必须有 [MOCK] 标注，不能假装是真实数据
- Mock 数据硬编码在 JS 中，不需要后端

### RAG 流程（Tab 3）
- 新闻数据：5-10条精心设计的 mock 新闻，覆盖茅台、宁德时代等热门股
- 检索算法：简化的关键词匹配（模拟向量检索效果）
- 生成：真实调用 LLM，基于 mock 新闻生成报告

### Agent 流程（Tab 5）
- Think/Plan/Act/Observe：前三步动画模拟，Observe + 结论真实 LLM 生成
- 工具调用：行情工具真实，估值/新闻工具返回 mock 数据
