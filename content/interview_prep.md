### 1.3 产品认知题

#### Q：你怎么理解 RAG 产品的核心设计挑战？

**回答框架（3 层）：**

1. **数据层**：数据质量和覆盖是 RAG 效果的天花板。需要解决多源异构数据接入、结构化存储、向量化策略（领域专用 Embedding vs 通用模型）。
2. **检索层**：金融领域语义特殊，通用检索效果差。需要设计同义词对齐（如 ROE=净资产收益率）、分层知识库（实时市场库+私有库）、召回精度评估方案。
3. **生成层**：幻觉和合规是核心风险。需要强制数据溯源标注、数值多源交叉验证、合规词检出机制。

**结合自身经历：** 我在恒生做研报平台时，就是按这三层来拆解问题的——从 10+ 数据源接入方案到金融同义词映射表，再到 RAG 召回结果的分级标注体系。

---

#### Q：PM 和算法工程师如何协作？你的经验是什么？

**回答要点：**
- 明确边界：PM 定义"What & Why"，算法定义"How"，但 PM 需要理解技术约束
- 在招股书提取项目中：我参与讨论切片与 Prompt 定位逻辑的协同设计，明确算法层（文本解析/切片）和 Prompt 层（字段提取/兜底）各自的处理边界
- 制定可验收的技术指标（准确率≥95%），让双方有共同的 done 定义

---

#### Q：Prompt Engineering 在产品中的价值是什么？你如何系统化它？

**回答要点：**
- 价值：在不微调模型的前提下，通过指令设计最大化 LLM 能力，成本低、迭代快
- 系统化方法：
  1. **模块化**：按功能拆分（文档定位、字段提取、格式规范、兜底逻辑）分别设计
  2. **复用化**：建立系统级金融同义词映射表供所有模块共用
  3. **可评估**：制定字段级准确率、溯源覆盖率等量化指标
  4. **迭代化**：Bad Case 入库 → 根因分析 → Prompt 修复 → 验证

### 1.4 产品设计题

#### Q：设计一个面向券商分析师的 AI 研报助手

**CIRCLES 框架：**

| 步骤 | 内容 |
|------|------|
| **Comprehend** | 理解场景：分析师面临多源信息碎片化、人工整合耗时、研报格式规范要求高 |
| **Identify users** | 核心用户：买方/卖方分析师；次要用户：研究部主管（质量把关）|
| **Report needs** | 需求：快速整合多源数据、生成结构化研报草稿、确保数据准确可溯源、符合合规要求 |
| **Cut through** | 优先级：数据准确性 > 生成效率 > 格式规范（金融场景准确第一）|
| **List solutions** | ① RAG 架构实现数据自动整合；② 双层知识库（实时+私有）；③ 分级溯源标注；④ 研报模板引擎 |
| **Evaluate** | 评估指标：研报采用率、基础修改轮次、数据引用准确率、合规词检出率 |
| **Summarize** | MVP 聚焦：数据整合 + 草稿生成 + 溯源标注，暂不做全自动发布 |

## 二、AI 产品设计题解题模板

### 通用解题步骤

```
1. 澄清问题（2 分钟）
   - 目标用户是谁？C 端还是 B 端？
   - 核心痛点是什么？
   - 有哪些约束？（合规、成本、技术）

2. 用户分析
   - 用户画像 + 核心使用场景
   - 用户旅程中的关键痛点

3. 产品方案
   - 核心功能设计（MVP 范围）
   - AI 能力选型（RAG / Prompt / Fine-tune）
   - 人机协作模式设计

4. 评估指标
   - 业务指标（效率提升、用户采用率）
   - 技术指标（准确率、召回率）
   - 合规指标（如果涉及）

5. 风险与挑战
   - 幻觉风险如何控制
   - 冷启动问题
   - 用户信任建立
```

### AI 产品设计常见坑

- **不要过度设计**：先问清楚 MVP 范围，别一上来就全自动化
- **合规先于效率**：金融、医疗场景，合规底线不能为了效率让步
- **量化评估体系**：必须给出可测量的成功指标，而非"提升用户体验"
- **人机协作模式**：说清楚 AI 做什么、人做什么、如何交接

## 三、竞品分析模板

### 3.1 分析框架

```
维度一：市场定位
- 目标用户群体
- 核心价值主张
- 定价策略

维度二：产品能力
- 核心功能矩阵
- AI 能力集成方式（RAG / Agent / Fine-tune）
- 数据处理能力

维度三：用户体验
- 关键用户旅程
- 交互设计亮点
- 已知痛点/差评

维度四：商业模式
- 收入来源
- 获客方式
- 护城河

维度五：技术路线
- 底层模型选型
- 数据飞轮建设
- 产品迭代节奏
```

### 3.2 金融 AI 产品竞品对比表（示例）

| 产品 | 目标用户 | 核心能力 | AI 技术路线 | 商业模式 | 优势 | 不足 |
|------|----------|----------|-------------|----------|------|------|
| 万得 AI | 机构投资者 | 数据终端 + AI 问答 | RAG + 私有数据 | 订阅制 | 数据壁垒深 | 交互体验差 |
| 同花顺 AI | 散户/机构 | 行情分析 + 研报 | LLM + 实时数据 | 广告+增值 | 用户基数大 | 专业深度弱 |
| 金融大模型（各厂自研）| 机构 | 合规问答/审批辅助 | Fine-tune 专有模型 | To B 定制 | 合规性强 | 通用性差 |
| 你的产品（恒生研报平台）| 卖方分析师 | 研报生成 + 溯源 | RAG + 双层知识库 | SaaS | 金融专业深度 | 用户规模小 |

### 3.3 竞品分析常用方法

- **功能矩阵对比**：列出核心功能，标注各家实现情况
- **用户差评挖掘**：AppStore/知乎/脉脉用户反馈 → 发现竞品痛点
- **定价策略分析**：理解商业逻辑，反推产品战略
- **技术路线推断**：通过产品表现反推底层 AI 架构选型
